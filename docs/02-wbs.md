## 3. 分解说明

本 WBS 的分解遵循 **结果导向** 原则，每个节点均代表一个可交付的成果或文档，不使用“开会、讨论、写代码、修改”等过程性动作。

- **一级交付物** 划分为四类：版本管理基础成果、项目管理文档成果、协作实践成果、总结报告成果。这覆盖了实验要求的全部交付范围，且边界清晰。
- **二级分解** 按照逻辑阶段或文档类型划分，例如“项目管理文档成果”下分为启动文档、WBS、进度计划、配置管理四个独立文档成果。
- **三级分解** 分解到可独立分配、可验证的最小成果单元，例如“在 GitHub 上创建仓库”是一个可完成的动作，“编写项目背景与目标”是一个可检查的文档片段。

**为什么这样划分？**
1. **版本管理基础成果** 确保团队在开始文档编写前拥有统一的协作基础设施（仓库、目录、首次提交）。
2. **项目管理文档成果** 是实验考核的核心，按文档类型拆分便于分配责任人和并行工作。
3. **协作实践成果** 覆盖冲突制造、分支合并等 Git 协作技能，这是实验的难点和亮点。
4. **总结报告成果** 整合全过程，体现团队反思和最终输出。
